Case study

Unity Explorer

Ground-up rebuild of Decentraland's primary Unity client, led while the product kept shipping. The web client had reached its limit: open-world requirements, more users and avatars in the same place, long sightlines and multi-threaded performance could not be solved by another patch. The rebuild created a cleaner foundation for world streaming, creator content, real-time presence and weekly delivery.

Unity / C# / Arch ECS / per-scene V8 JavaScript runtimes / CRDT protocol / LiveKit / Addressables & asset bundles / Sentry

RepositoryAll projects

Context

Decentraland needed the client to support a more seamless open-world experience: more people and avatars in the same place, long sightlines, real-time social presence, creator content and weekly delivery. The old web client had carried years of technical debt. It could still be patched, but every patch made something else more fragile: more special cases, weaker ownership, slower development and less confidence in releases.

The decision was to rebuild because upkeep had stopped being a realistic option. This was not "a Unity rewrite" for its own sake. It was a client architecture reset so the product could move toward an open world without making every future change harder.

What changed

  • Replaced an architecture that could not sustainably support the open-world direction with a modular Unity client designed around streaming, observability, creator content and release confidence.
  • Moved performance work toward the product requirement: more users and avatars in the same place, longer sightlines and smoother frame times.
  • Moved the engineering scoreboard toward stability, crash reduction, performance, multiplayer reliability and release quality.
  • Connected telemetry to prioritization: crashes, hangs, frame time, loading and release regressions became inputs to technical direction.
  • Improved release discipline so the team could ship weekly without treating every release as a special event.
  • Created clearer ownership across client systems, runtime integration, content loading, QA and distribution.

Three decisions define the client. Game state lives in a data-oriented ECS because frame-time spikes are user-facing product problems. Creator scenes run in isolated V8 runtimes and communicate through CRDT, which helps reconcile scene and engine state but does not solve multiplayer authority by itself. Real-time multiplayer still needs an explicit authority boundary; peer-to-peer did not scale as more players shared the same space, so reliable voice, avatar sync and coordination moved toward server-backed infrastructure.

Interesting problems

  • How does one bad scene not take down the client? Scenes are isolated per V8 engine with explicit error budgets — a scene that throws too many errors per minute is suspended, not the client. User-generated content means the client must assume hostile input as the normal case.
  • How do you keep a frame budget with unbounded content? No allocation in hot paths, aggressive pooling, LOD and prioritization of avatars by relevance — the work is bounded even when the world is not.
  • How do you rebuild without stopping delivery? Split the work into shippable verticals and keep release quality visible. The legacy client kept serving users while the new one earned its place.
  • How do you make performance work measurable? Tie technical debt to product KPIs — loading time, frame rate, crash rate — and let telemetry, not opinion, order the backlog.
  • Where does authority live in real-time multiplayer? CRDTs are useful for state reconciliation, but they do not remove the need to decide who owns the truth. Peer-to-peer looked attractive, but it became error-prone as more players joined the same space; connections failed and state resolution became too fragile. The practical answer was to keep the protocol open while using server-backed multiplayer where reliability required it.

Lessons learned

  • A rebuild succeeds only if it changes operational behavior, not just code structure.
  • CRDTs help with state reconciliation, but they are not a complete multiplayer authority model.
  • Peer-to-peer multiplayer stops being elegant when player density turns connection and state resolution into product risk.
  • Telemetry is a prioritization tool, not a dashboard decoration.
  • For a social product, stability and presence are product features.
