Case study

AI for Engineering Teams

Putting AI where it changes engineering decisions: review signal, QA evidence, knowledge access and leadership visibility.

AI PR review / visual regression suites / per-PR performance benchmarks / scheduled analysis agents / engineering dashboards

Runs in production on an open-source codebase.All projects

Problem

Most AI adoption in engineering optimizes for visible activity: more code generated, more text summarized. That is not the hard part. The hard part is putting AI where it changes outcomes: fewer regressions escaping, less dependency on individual memory, and decisions made from evidence rather than recollection.

The systems below run in production on an open-source codebase, so they are inspectable rather than aspirational.

The systems

AI review on every PR

Each pull request carries an AI review status check that analyzes correctness, performance and error handling and sets pass/fail. It does not replace the accountable reviewers — every PR still needs developer and QA approval — it raises the floor before humans spend attention.

Visual regression as a comment

A PR comment triggers a pixel-diff suite against committed image baselines and posts the verdict back to the PR; a second command regenerates baselines when a change is intentional. Rendering regressions became a reviewable artifact instead of a QA lottery.

Performance signal per PR

Successful builds dispatch frame-time benchmarks on a dedicated machine fleet, and the results land back on the PR as a comment — CPU, GPU, exceptions. A performance regression is visible before merge, not after release.

Leadership intelligence agents

Scheduled agents turn repository activity into a daily engineering dashboard: releases and deltas, CI health, escaped-defect and regression rates, review distribution and bus factor per system — with pinned metric definitions so week-over-week deltas mean something.

Interesting problems

  • Where should AI sit in the review process? Before human attention, never instead of it. The AI check runs first and cheaply; approval authority stays with named developers and QA. The moment "the AI passed it" becomes an excuse, the system is net negative.
  • How do you avoid AI becoming noise? Make it decision-shaped. A pass/fail status, a pixel diff, a frame-time delta — each output either changes what happens to the PR, or it gets removed. Commentary without a decision attached is overhead.
  • How do you trust AI-computed metrics? Pin the definitions. Every dashboard metric has a fixed formula and window; when a definition changes, the change is called out so a delta is never mistaken for a trend.
  • How do you make knowledge access safer? Source-backed answers with visible uncertainty beat confident summaries. The dashboard links to the PRs, issues and releases it counted, so any number can be re-derived by a human.

Lessons learned

  • AI adoption only matters when it changes how decisions are made.
  • Quality and knowledge access are better first-order goals than raw speed.
  • The useful workflows are the ones boring enough to run every week.
