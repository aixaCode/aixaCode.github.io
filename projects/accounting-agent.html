Architecture case study

AI Finance Automation

I built two small finance systems for my own accounting and project budgets. An LLM reads documents and suggests matches; I confirm them, and ordinary Python calculates every number.

Python / model-agnostic LLM document extraction / bank-statement CSV parsing / deterministic rules engine / statement-derived FX / JSON caches / HTML dashboard / Proxmox LXC

Internal/private system. Implementation details are based on local project code.All projects

Overview

This started with a recurring annoyance: statements, invoices and project budgets were easy to lose track of. I wanted help with the repetitive document work, but I did not want an AI agent deciding what went into the accounts. It reads and suggests. The ledger, tax maths and filing rules remain deterministic.

Two sibling systems share that boundary. The accounting agent parses period-scoped bank-statement CSVs, converts currencies to GBP, categorises transactions through a rules engine, reconciles them against filed invoice PDFs, computes P&L and versioned finance/tax estimates, and renders an HTML dashboard plus a privacy-safe summary JSON. The project budget agent runs the same pattern for project spend: PDFs arrive in an inbox, an LLM extracts fields and suggests a project, a human confirms or overrides, and deterministic code aggregates budget burn and cost split.

Both deploy as unprivileged LXC containers on a Proxmox host, next to the personal assistant that consumes their summaries read-only.

Requirements

TrustSuggestions must never move money until confirmed.
ReproducibilitySpend, remaining budget and category split must be recalculable from local state.
PrivacyStatements, invoices, caches and dashboards are git-excluded; the repo itself is local-only Git with no remote.
Agent boundaryThe personal assistant reads only aggregate JSON, not source invoices.

The AI boundary is visible in the graph: the LLM appears only where unstructured text needs interpretation — reading PDFs and suggesting matches — and everything downstream of a suggestion is deterministic. The sibling budget agent follows the same shape with an explicit human confirmation step between suggestion and aggregation (extract → assign → confirm → aggregate), driven by a projects.yaml budget registry.

corrections.pySanctioned manual fixes for misfiled, mismatched or double-counted invoices.Pure file and cache surgery — no AI calls and no accounting maths in the module.

Implementation detail

The part I care about most is the confirmation step. In the budget agent, the aggregator ignores assignments I have not confirmed and reports a warning when the data is ambiguous:

for key, assignment in assignments.items():
    if not assignment.get("confirmed", False):
        continue

    project = assignment.get("project", "unknown")
    amount = extractions.get(key, {}).get("amount")

    if amount is None:
        warnings.append(f"{key}: no amount extracted - excluded from spend")
        continue

The same discipline runs through the accounting agent: tax estimates are versioned so every run is comparable with the last, cache writes are atomic, and the corrections module's only powers are moving files and migrating cache keys.

Interesting engineering problems

  • Where should AI stop? AI extracts fields from PDFs and suggests matches or project assignments. It does not approve spend, compute totals, categorise the ledger or touch finance maths.
  • How do you convert currency without an FX feed? Month-level rates are derived from the statements' own conversion rows; missing months fall back to the latest prior rate, then to an explicit configured approximation — never a silent guess.
  • How do you keep AI cost and drift low? The matcher makes one batched call per run and caches per transaction, so a re-run asks nothing it already knows.
  • How do you fix mistakes without corrupting state? Corrections are deterministic file moves and cache-key migrations, with duplicate and bundled charges detected by exact arithmetic — hand-editing the caches is never the fix.
  • How do you avoid quiet finance errors? Only confirmed assignments count toward spend, extraction caches invalidate on file metadata, JSON writes are atomic and data-quality checks suggest instead of mutate.
  • How do agents use the data safely? The personal assistant mounts a single aggregate summary read-only, so it can answer finance questions without access to statements or invoice PDFs.

Trade-offs

DecisionWhyCost
Local files over SaaS accounting integrationsLower privacy risk and easier iteration for a private workflow.More local operations and less automatic bank reconciliation.
JSON caches over a databaseSimple, inspectable and enough for the current volume.Will need migration if workflow grows into multi-user operations.
Manual confirmation in the middle of the pipelinePrevents AI suggestions from becoming financial truth.Not fully hands-off, intentionally.
Local-only Git with no remote and a pre-push hookFinancial data cannot leave the machine, even by habit.No off-site history; backups belong to the storage layer instead.

Lessons learned

  • AI is useful for reading untidy documents. It should not quietly turn a suggestion into an accounting decision.
  • I trust small, inspectable scripts more than a general agent with broad permissions.
  • For a workflow I run myself, a private dashboard is often enough. It does not need to become a product.
