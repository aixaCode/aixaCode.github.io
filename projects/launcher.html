Case study

Launcher & Distribution Platform

Rust rework of the desktop Launcher: installer, updater, authentication and release path for getting a desktop client into users' hands reliably.

Rust / Windows / installer / updater / authentication / telemetry / Epic distribution

Internal or private project material.All projects

Problem

A desktop app is not shipped when the build compiles. The Launcher rework moved the install/update path into a more reliable Rust-based foundation so users can install, authenticate, update, recover from failures and trust the distribution path.

Goals

  • Improve install and update reliability
  • Support signed builds and distribution channels
  • Connect launcher behavior to telemetry
  • Reduce user drop-off before the first experience
  • Keep hotfixes small and release paths predictable

Architecture

graph TD
  Build[Client build] --> Sign[Code signing]
  Sign --> Package[Installer / updater]
  Package --> Channels[Website / Epic / desktop channels]
  Channels --> Launch[User launch]
  Launch --> Telemetry[Install + launch telemetry]
  Telemetry --> Fixes[Release fixes]

Interesting problems

  • How do you debug a user who never reaches the app? Launcher telemetry has to cover install, update, launch and authentication paths.
  • How do you keep releases fast without making distribution risky? Separate release gates from big process and keep hotfix scope minimal.
  • How do you treat distribution as product work? First-run success is part of onboarding, not only DevOps.

Lessons learned

  • Launcher reliability is funnel work. If it fails, the product never gets a chance.
  • Distribution bugs should be classified with the same seriousness as client crashes.
  • The smallest release process that catches real risk is usually the best one.
