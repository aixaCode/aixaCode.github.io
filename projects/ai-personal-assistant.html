Architecture case study

AI Assistant

A personal AI assistant with separate work and personal profiles, persistent context, calendar/email ingress and policy-based tool execution.

TypeScript / Node.js / Claude Agent SDK / provider-agnostic model interface / MCP server / Telegram / email / Google Calendar / Markdown knowledge base

SourceAll projects

Overview

I built this because generic chat loses the context that makes an assistant useful: project notes, calendar commitments, work vs personal boundaries, local files, and the repeated operational tasks that do not belong in my head. The assistant is designed for planning, knowledge lookup, reminders, calendar-aware work and controlled local automation.

The model is a replaceable component, not the architecture. Today it runs Claude through the Agent SDK; a provider-agnostic model interface sits beside it so a local model can drive the same tool registry and permission gate without changing the security model.

Requirements

  • Run locally on a developer-controlled machine, with private workspace data outside the Git repository.
  • Separate personal and work environments with different tools, directories and model settings.
  • Use Claude through the Agent SDK while preserving a future seam for local model runtimes.
  • Execute tools only through deterministic authorization, not through prompt instruction alone.
  • Support CLI, private Telegram, email ingress and calendar-aware workflows without losing conversation continuity.
  • Produce an auditable execution history for tool decisions and confirmations.

The model receives context and may request tools through the local MCP server. It does not directly execute privileged actions. Every tool request passes through metadata, profile policy, path validation and optional human confirmation before execution. The dotted line is the model boundary: the runtime is the only component that knows which model is running, so swapping Claude for a local runtime changes nothing on the tool or permission side.

Key components

Runtime adapter

src/runtime.ts wraps the Claude Agent SDK, injects the local MCP server, restricts allowed tools to the active profile and supports session resume.

Profile isolation

src/profiles.ts defines each profile's workspace directory, model and tool list. Work and personal contexts do not share tools by default.

Context builder

src/context.ts constructs system text from profile context, current time, memory files and the knowledge-file index before model invocation.

Permission engine

src/permissions.ts evaluates tool metadata, asks for human confirmation when required and logs allow/deny decisions.

Safe file layer

src/files.ts blocks traversal and symlink escapes, confines writes to notes/ and knowledge/, and keeps private workspace data out of Git.

Telegram state

src/telegram-state.ts persists session IDs, serializes conversations per user and lets confirmation replies bypass the normal queue.

Model seam

src/models/model.ts defines a provider-agnostic chat interface — messages and tool definitions in, text or tool calls out — so a local runtime can replace Claude without touching tools or permissions.

Run queue

src/queue.ts is a bounded FIFO that caps concurrent agent runs to stay inside subscription limits; deliberately no priorities or retries until a real need shows up.

Implementation detail

Tool authorization is a typed runtime decision. The language model can propose an action, but the runtime decides whether that action is allowed.

export const toolMeta = {
  read_file: { scope: "read_local", risk: "auto" },
  write_note: { scope: "write_local", risk: "confirm" },
  calendar_create_event: { scope: "write_external", risk: "confirm" },
} as const;

export async function enforce(profile, confirm, name, input) {
  const meta = toolMeta[name];
  if (!meta) {
    audit({ profile: profile.name, tool: name, decision: "deny", reason: "unknown tool" });
    return false;
  }
  if (meta.risk === "confirm") {
    const ok = await confirm(formatConfirmation(profile, name, input));
    audit({ profile: profile.name, tool: name, scope: meta.scope, decision: ok ? "allow" : "deny" });
    return ok;
  }
  audit({ profile: profile.name, tool: name, scope: meta.scope, decision: "allow" });
  return true;
}

Filesystem access uses the same principle. Path resolution is lexical first, then checked against real paths so symlink escapes are rejected. Writes are limited to explicit knowledge and notes directories.

Model portability is enforced by an interface, not an aspiration. Anything that can answer chat() with text or tool calls can drive the assistant through the same registry and permission gate:

export interface Model {
  chat(input: { messages: Message[]; tools: ToolDefinition[] }): Promise<{
    text?: string;
    toolCalls?: ToolCall[];
  }>;
}

Interesting engineering problems

  • Policy-based tool execution. The model generates intent, never authority. Tool calls are translated into named commands evaluated against profile allowlists, scope/risk metadata, confirmation requirements and audit logging.
  • Prompt injection boundary. The dangerous boundary is not the prompt; it is the moment text becomes action. Unknown tools are denied, risky tools ask the human and built-in SDK tools are excluded by the profile allowlist.
  • Memory architecture without pretending everything is vector search. Durable context is split into profile context, Markdown knowledge, append-only memory notes and available-file indexes. Semantic search and SQLite are intentionally future tiers, not hidden dependencies.
  • Profile isolation. Personal and work profiles have separate roots, model settings and tool sets. File validation enforces workspace boundaries in code, including traversal and symlink checks.
  • Asynchronous UX. Telegram requests are serialized per user so runs do not trample each other, while confirmation responses bypass the queue to unblock active tool calls.

Trade-offs

DecisionTrade-offWhy
Claude Agent SDK instead of owned model loopLess control over the model loop, faster path to a useful assistantThe goal was a working local assistant on an existing subscription, with a model interface kept for later local runtimes.
Markdown knowledge before vector searchLess fuzzy retrieval, better debuggabilityFor early usage, grep-able source-of-truth files were more reliable than opaque embeddings.
Explicit profile tool listsMore configuration work, stronger isolationProfiles become the authorization boundary instead of relying on model behavior.
Human confirmation for write/external actionsSlower automation, safer executionThe assistant should be useful before it is autonomous.

Results

  • CLI and private Telegram ingress working with durable session resume.
  • Personal and work profiles isolated by workspace root and tool allowlist.
  • Permission gate, safe file layer and append-only audit log implemented in code.
  • Google Calendar, email, local knowledge, reminders, work notes and skill-loading tools integrated behind the same authorization path.
  • Security-focused tests cover configuration, permissions, safe files, calendar, email and Telegram state.
  • Provider-agnostic model interface in place with local-runtime adapters stubbed (Ollama/vLLM); Claude via the Agent SDK is the wired runtime today.

Lessons

  • Security boundaries belong in deterministic code, not in prompt wording.
  • The model should propose actions; the runtime should authorize them.
  • Useful memory starts with clear information architecture. Vector search is a tool, not the foundation.
  • Local-first assistants need operational features early: audit logs, session persistence, bounded concurrency and recovery paths.
