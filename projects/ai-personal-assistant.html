Architecture case study

AI Assistant

A self-hosted personal operations assistant that turns messages, email and documents into durable knowledge, commitments, reminders and human-approved actions.

TypeScript / Node.js / Claude Agent SDK / SQLite / MCP server / Telegram / email / Google Calendar / Markdown knowledge / NAS-backed recovery

SourceAll projects

Overview

I built this because I kept repeating the same context in generic chat: current projects, calendar commitments, where files lived and whether I was asking about work or home. I wanted an assistant that could retain that context and help with planning, reminders and a few repetitive local tasks.

The project has evolved from a conversational tool runner into a durable, event-driven assistant. Telegram and email are intake channels; accepted work is queued before acknowledgment, retried after transient failures and converted into structured operational state as well as human-readable knowledge.

It currently runs Claude through the Agent SDK, routing routine work to Sonnet and complex reasoning or documents to Opus. The model boundary remains separate from tools and permissions, so a local runtime can be added later without rebuilding the action layer.

Requirements

  • Run locally on a developer-controlled machine, with private workspace data outside the Git repository.
  • Separate personal and work environments with different tools, directories and model settings.
  • Use Claude through the Agent SDK while preserving a future seam for local model runtimes.
  • Execute tools only through deterministic authorization, not through prompt instruction alone.
  • Support CLI, private Telegram, email ingress and calendar-aware workflows without losing conversation continuity.
  • Produce an auditable execution history for tool decisions and confirmations.
  • Accept concurrent messages durably and recover unfinished work after model limits, process restarts or infrastructure failure.
  • Separate curated knowledge from operational state such as tasks, commitments, people, decisions and unresolved questions.
  • Retain provenance, confidence and verification dates for learned facts instead of treating every model inference as equally trustworthy.

Incoming messages are acknowledged only after durable acceptance. Telegram is the default control and delivery surface; email sends a receipt, while analysis and suggested reply drafts return privately to the mapped user in Telegram. Temporary failures remain queued and retry automatically, including at the reset time reported by a model subscription limit.

The model may request tools through the local MCP server, but it does not directly execute privileged actions. Every request still passes through profile policy, tool metadata, path validation and optional human confirmation. Storage is intentionally split: Markdown remains inspectable knowledge, SQLite holds queryable operational objects, and source documents preserve the evidence behind both.

Key components

Runtime adapter

src/runtime.ts wraps the Claude Agent SDK, injects the local MCP server, restricts allowed tools to the active profile and supports session resume.

Profile isolation

src/profiles.ts defines each profile's workspace directory, model and tool list. Work and personal contexts do not share tools by default.

Context builder

src/context.ts constructs system text from profile context, current time, memory files and the knowledge-file index before model invocation.

Permission engine

src/permissions.ts evaluates tool metadata, asks for human confirmation when required and logs allow/deny decisions.

Safe file layer

src/files.ts blocks traversal and symlink escapes, confines writes to notes/ and knowledge/, and keeps private workspace data out of Git.

Durable intake

src/durable-jobs.ts persists Telegram messages, PDFs and email before processing. It deduplicates message IDs, recovers interrupted jobs and applies bounded retry with subscription-reset awareness.

Operational state

src/operational-state.ts stores queryable tasks, people, projects, decisions, commitments, reminders and unresolved questions separately from narrative knowledge.

Knowledge provenance

Learned facts retain their source, timestamp, confidence, scope, verification date and whether they were explicitly stated or inferred, reducing the risk of stale guesses becoming truth.

Document pipeline

Email and Telegram PDFs are read by the model, meaningfully renamed and organised under a taxonomy that mirrors knowledge. Accounting documents are handed to a separate accounting workflow.

Telegram state

src/telegram-state.ts persists sessions and reminders, serializes each conversation and lets confirmation replies unblock an active tool call. Shared scheduled work executes once before delivery to multiple recipients.

Model seam

src/models/model.ts defines a provider-agnostic chat interface — messages and tool definitions in, text or tool calls out — so a local runtime can replace Claude without touching tools or permissions.

Model routing

Routine requests and scheduled work use Sonnet; complex reasoning and document analysis use Opus. A bounded execution queue limits concurrent model runs independently of the durable intake queue.

Recovery

Local SQLite is authoritative during execution. Serialized snapshots are stored on the NAS, while an LXC-local workspace fallback lets Luke continue during a NAS outage and sync changes back after recovery.

Implementation detail

Tool authorization is a typed runtime decision. The language model can propose an action, but the runtime decides whether that action is allowed.

export const toolMeta = {
  read_file: { scope: "read_local", risk: "auto" },
  write_note: { scope: "write_local", risk: "confirm" },
  calendar_create_event: { scope: "write_external", risk: "confirm" },
} as const;

export async function enforce(profile, confirm, name, input) {
  const meta = toolMeta[name];
  if (!meta) {
    audit({ profile: profile.name, tool: name, decision: "deny", reason: "unknown tool" });
    return false;
  }
  if (meta.risk === "confirm") {
    const ok = await confirm(formatConfirmation(profile, name, input));
    audit({ profile: profile.name, tool: name, scope: meta.scope, decision: ok ? "allow" : "deny" });
    return ok;
  }
  audit({ profile: profile.name, tool: name, scope: meta.scope, decision: "allow" });
  return true;
}

Filesystem access uses the same principle. Path resolution is lexical first, then checked against real paths so symlink escapes are rejected. Writes are limited to explicit knowledge and notes directories.

Model portability is enforced by an interface, not an aspiration. Anything that can answer chat() with text or tool calls can drive the assistant through the same registry and permission gate:

export interface Model {
  chat(input: { messages: Message[]; tools: ToolDefinition[] }): Promise<{
    text?: string;
    toolCalls?: ToolCall[];
  }>;
}

Interesting engineering problems

  • Policy-based tool execution. The model generates intent, never authority. Tool calls are translated into named commands evaluated against profile allowlists, scope/risk metadata, confirmation requirements and audit logging.
  • Prompt injection boundary. The dangerous boundary is not the prompt; it is the moment text becomes action. Unknown tools are denied, risky tools ask the human and built-in SDK tools are excluded by the profile allowlist.
  • Knowledge is not workflow state. Human-readable Markdown remains the source for narrative context, while SQLite stores lifecycle-bearing objects that need filtering, status transitions and due dates. Facts in either layer carry explicit provenance.
  • Profile isolation. Personal and work profiles have separate roots, model settings and tool sets. File validation enforces workspace boundaries in code, including traversal and symlink checks.
  • Durable asynchronous UX. Ingress and execution are separate. Luke acknowledges accepted work immediately, serializes conversations, retries recoverable failures and avoids asking the user to resend a request.
  • One computation, many recipients. Shared scheduled routines execute once and fan out the same result, avoiding duplicate model runs and inconsistent summaries.

Results

  • CLI, private Telegram and allow-listed email intake working with durable session resume and persistent request queuing.
  • Personal and work profiles isolated by workspace root and tool allowlist.
  • Permission gate, safe file layer and append-only audit log implemented in code.
  • Google Calendar, Telegram-first email drafts, structured reminders, accounting intake, document organisation and skill-loading tools integrated behind the same authorization path.
  • Queryable operational objects and provenance-aware facts separate active commitments from narrative knowledge.
  • Concurrent requests, subscription limits and restarts no longer require users to resend accepted work.
  • Shared scheduled routines execute once and fan out to all intended recipients.
  • Security and reliability tests cover configuration, permissions, safe files, calendar, email, durable jobs, operational state and Telegram state.
  • Provider-agnostic model interface in place with local-runtime adapters stubbed (Ollama/vLLM); Claude via the Agent SDK is the wired runtime today.

Lessons

  • A prompt is not a security boundary. Permission checks have to live in code.
  • The model can suggest an action, but the runtime decides whether it is allowed.
  • Good folder structure and clear profile notes solved more of the memory problem than vector search would have.
  • Audit logs and session recovery became necessary much earlier than I expected.
  • Acknowledgment must mean durable acceptance, not merely that a webhook or poller saw a message.
  • Personal-assistant memory needs provenance and lifecycle, not just better retrieval.

Next expansion

The next step is a proactive follow-up engine. Luke already has intake, structured commitments, reminders, source documents and reply drafting; the missing layer is lifecycle management across them.

The engine will connect incoming requests to people and projects, track what I promised and what I am waiting for, detect later evidence that an item was resolved, and surface only genuinely open follow-ups. Telegram actions such as Mark done, Remind tomorrow, Create draft, Change due date and Show source keep the human in control.

After that, a small private operations dashboard can expose queue health, failed work, open commitments, reminders, stale or low-confidence facts and recent assistant activity. Telegram remains the conversational interface; the dashboard becomes the inspection and correction surface.
